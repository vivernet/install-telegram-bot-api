# Предложенные задачи по итогам быстрого ревью

## 1) Исправление опечатки (README)

**Проблема:** в разделе «Процесс обновления» используется форма «Перезапуск службу», что является грамматической опечаткой.

**Что сделать:** заменить на «Перезапуск службы».

**Почему важно:** даже мелкие языковые ошибки снижают доверие к документации и усложняют восприятие.

**Критерий готовности:** в `README.md` исправлена формулировка, и в этом разделе нет грамматических ошибок.

---

## 2) Исправление ошибки в коде (проверка порта)

**Проблема:** результат `port_check` теряется из-за конструкции `pidinfo=$(port_check "$API_PORT" "$SERVICE_NAME" || true)`; из-за `|| true` код возврата всегда становится `0`, и ветка обработки занятого порта фактически не срабатывает.

**Что сделать:** переписать вызов так, чтобы сохранить реальный exit code функции `port_check` (например, временно отключать `set -e` вокруг вызова или использовать иной безопасный паттерн без `|| true`).

**Почему важно:** сейчас возможен запуск при занятом порте с некорректной логикой дальнейших шагов.

**Критерий готовности:** при занятом порте не «нашей» службой скрипт предлагает сменить порт/прервать установку, а при свободном порте продолжает установку без ложноположительных сообщений.

## 3) Исправление комментария/документации (несоответствие)

**Проблема:** комментарий перед блоком отключения службы говорит «если существует и запущена», но в коде внутри сначала проверяется `unit_is_enabled`, а не факт запуска (`unit_is_active`), что вводит в заблуждение.

**Что сделать:** привести комментарий в соответствие с реальным поведением (либо исправить комментарий, либо изменить логику, чтобы она соответствовала формулировке).

**Почему важно:** точность комментариев критична для безопасного сопровождения инсталляционного скрипта.

**Критерий готовности:** комментарий и проверяемые условия (`enabled`/`active`) совпадают по смыслу.

---

## 4) Улучшение тестов

**Проблема:** в репозитории отсутствуют автоматические тесты для критичных функций shell-скрипта (`port_check`, `read_config_value`, `detect_default_branch`).

**Что сделать:** добавить минимальный набор Bats-тестов с моками внешних команд (`ss`, `systemctl`, `git`) и отдельный CI-шаг для запуска тестов.

**Почему важно:** скрипт управляет systemd, портами и конфигами — регрессии здесь дорого обходятся.

**Критерий готовности:**
- есть тесты минимум для трёх сценариев `port_check` (свободен / занят нашей службой / занят чужим процессом);
- есть тесты для чтения параметров из конфигурации;
- тесты запускаются в CI и падают при нарушении ожидаемого поведения.
